# AIGC 图片生成与编辑 — 用户可完成型 Agent 交互方案 v0.1

**读者**：产品经理、体验设计（中文为主，尽量少堆术语）  
**范围**：买家店定制流程里的「AI 生图 + 选用 + 进画布」；不写具体 bug 修复清单，只服务交互与验收口径。  
**真源**：API 契约以 `docs/AIGC-API-MVP-SKELETON-v2.md` 为准；本文件为**交互层单一真源**（与 `AIGC-STOREFRONT-GAP-v1.md` 互补：后者偏工程 gap，本文件偏用户侧流程）。

## 1. 现状摘要

### 1.1 谁负责什么（三层分工）

| 角色 | 负责什么 |
|------|----------|
| **浏览器里的定制页（CustomizeEditor）** | 收集：文案、参考图文件、是否多图模式、候选张数；先调上传接口拿 `asset_id`，再调创建任务接口；轮询任务状态；展示候选缩略图与确认倒计时；用户点「确定」后把选用图的 URL **作为新图层**插进画布。 |
| **Next 里的 AIGC API（BFF）** | `POST /api/aigc/reference-assets` 收图 → 返回 `asset_id`；`POST /api/aigc/generations` 建任务（通常先 `queued`）；`GET .../generations/[jobId]` 返回状态与候选；`POST .../confirm` 锁定一张候选。任务持久化、队列消费见骨架文档（有库走表 + Cron，无库走内存 + 轮询懒推进）。 |
| **Provider（模型供应商适配层）** | `lib/aigc-provider.ts` 按环境选择 **mock / fal / ttapi / dashscope** 等实现；把「网关统一字段」翻译成各厂商 HTTP；返回候选图 URL 与 **warnings**（例如 mock 说明、多参考降级提示）。 |

### 1.2 买家侧界面现状（CustomizeEditor）

- **入口**：`components/CustomizeEditor.tsx` 右侧「AI 画像（MVP デモ）」卡片。
- **模式**：由前端逻辑推断 — 无参考图 → **文生图（txt2img）**；有参考图且未勾选「多图组合」→ **单参考图生图（img2img）**；勾选且 ≥2 张 → **多参考（multi_ref）**，`references` 里每张图按顺序轮换 `subject / style / layout / other` 角色（用户**不能**逐张改角色）。
- **数据流**：本地选文件 → `POST /api/aigc/reference-assets`（multipart，`files`）→ 得到 `reference_asset_ids` → `POST /api/aigc/generations`（JSON：`product_id`、`mode`、`prompt`、`negative_prompt`、`candidate_count`、可选 `reference_asset_ids` / `references`）→ 同步轮询 `GET .../[jobId]` 直到 `ready` / 失败 / 超时 → 用户选缩略图 → `POST .../confirm` → 把 `selected.url` 写成**新图片图层**（插在图层列表前面并自动选中）。
- **与定制保存的关系**：确认落画布后，提示用户仍需「デザインを保存」走 `POST /api/customizations`；**确认接口本身不落定制单**（与骨架文档一致）。

### 1.3 服务端与 Provider 一句话

- **generations**：创建异步任务 → worker/provider 填 `candidates` → 用户在 `confirm_deadline_at` 前 confirm。  
- **reference-assets**：把用户图变成带 `asset_id` 的短期可用引用（可配 Blob 存公网 URL，供 worker 拉像素）。  
- **TTAPI 等**：网关只约定「模式 + 参考列表 + 文案」；TTAPI 实现里文生图走 Flux `generate`、带参考走 `edits`（多图 multipart）；真实「多图语义融合」深度受所选 **flux 型号与厂商参数** 限制，产品侧要预期 **warnings** 与降级说明。

## 2. 问题与差距（抽象成产品问题）

以下对照常见验收反馈与代码/文档事实，抽象为**交互与认知**问题（不单点列 bug）。

1. **能力边界不清**：同一面板里塞了文生图、单图编辑、多图融合；仅靠一个小勾选区分，**用户不知道当前这一次请求到底走哪条管线**，容易误开 multi_ref 或反过来期待「多图自动合成」却未勾选。
2. **「角色」不可控**：multi_ref 下角色按文件名顺序轮换分配，**用户没有为每张图选 subject/style 的检查点**，与 PRD 里「多图 + 文案」的可理解组合有差距。
3. **编辑器心智分裂**：PRD 原文曾写 MVP「无画布」（`PRD-FUNDAMENTAL-v0.1.md` §G2/§8.4 等）；现买家店已有**图层画布** + AIGC 侧栏。用户会期待「像专业工具一样编辑」，但实际 AIGC 仅是**侧栏一块**；**能力弱**的感受常来自预期与 MVP 范围不一致，而非单按钮问题。
4. **缺少显式「任务步骤」**：从上传到出图是一键「候補を生成」，中间没有**可读的步骤条或任务卡**，失败/超时时用户不知道应回到哪一步重来。
5. **迭代闭环弱**：「基于上一张再改」在数据上可通过再上传参考图实现，但界面**没有引导**把「已确认候选 / 画布选中层」一键变成下一步的参考，用户要自己导出再选文件，**摩擦大、易放弃**。
6. **保存与成功反馈**：历史上存在「画布已更新但保存状态文案误导」类反馈；交互上需要把 **「生图确认」** 与 **「定制保存到服务器」** 分成两个可理解的检查点（当前文案已有部分提示，方案里强化为固定步骤用语）。

## 3. 「用户可完成的 Agent」在本产品里的定义

**在本定制场景，不指**黑盒自动替用户做完所有设计、也不指后台 n8n 里无人值守的长链路。

**指的是**：

- **任务被拆成可见步骤**（例如：选目标 → 备素材 → 自检清单 → 生成 → 选结果 → 落画布），用户**知道自己卡在哪一步**。
- **每一步有明确输入与输出**：输入是用户确认过的文案/图/模式；输出是候选图或明确错误原因，而不是静默改参数。
- **不可逆动作前必有检查点**：例如「多图融合」前展示每张图的角色与缩略图；「替换画布某层」前二次确认（若未来开放替换）。
- **系统不替用户做语义合并决策**：不把多种意图混在一个「一键智能」里；若模型侧有降级，用 **warnings** 告知，而不是假装按理想语义执行完。

### 与 n8n / 后台 Agent 的边界

| 层级 | 做什么 | 不做什么 |
|------|--------|----------|
| **买家侧（本方案）** | 轻量 **向导 / 任务卡**：收集参数、展示检查点、调用已有 HTTP API、展示候选与倒计时。 | 不在浏览器里跑复杂多节点编排；不把订单/履约逻辑交给「前端 Agent」。 |
| **n8n / 后台** | 运营自动化、批量生图、跨系统同步等（见仓库 `AGENTS.md` 与后台工具）。 | 不替代买家「选用与确认」的合规与心智步骤；买家路径仍以 **BFF + 用户显式确认** 为主。 |

## 4. 目标交互模型

### 4.1 主方案：**「任务卡 + 检查点」式 Guided Copilot（推荐）**

**一句话**：用户先**选任务类型**（一张任务卡），再按卡片上的步骤填表；每步可返回修改；生成与选图仍是现有 API，只是**包装成可走完的流程**。

#### 用户旅程（6～8 步）

1. **选任务**：在侧栏顶部选四类之一 — 「只用文字出新图」「用一张图改款」「多张图按角色一起参与」「基于上一张继续改」（最后一类若画布无可用图则灰显并说明原因）。
2. **备素材**：按任务类型展示不同表单 — 文生图仅文案；单参考要求 1 张上传或「使用画布选中图层」；多参考要求 ≥2 张且**逐步为每张指定角色**（subject / style / layout / other）；迭代类默认带「上一张输出」缩略图，可再加一张补充参考。
3. **自检清单（检查点）**：只读摘要 — 模式名、提示词摘要、参考张数与角色表、候选张数；用户点「参数无误，去生成」才解锁生成按钮（避免误触）。
4. **生成与等待**：调用现有 `reference-assets`（如需）→ `generations`；展示 `queued` / `processing` 文案，可展示服务端 `warnings`（不遮挡主状态）。
5. **选候选**：与现有一致 — 缩略图网格 + 倒计时；可选「全部不满意，返回第 2 步改参数」。
6. **落画布（分支）**：  
   - **默认**：新增一层（与当前行为一致）。  
   - **可选**：「替换当前选中的图片层」（需确认对话框；MVP 可只做新增，此项标 P1）。  
   - **蒙版区域编辑**：占位 — 界面预留「未来：只改选中区域」说明，不调 API（P1+）。
7. **保存提醒**：固定 copy —「图已进画布；请点保存定制才会进购物车链路」— 与 `confirm` 不写库的事实对齐。

#### 每步界面要素（主方案）

| 步骤 | 主要控件 | 分支 |
|------|----------|------|
| 选任务 | 四个任务卡（单选） | 迭代任务在无选中/无可引用层时禁用 |
| 备素材 | 文件选择 / 画布引用按钮、多图时「每张一行：缩略图 + 角色下拉」 | 文生图隐藏参考区 |
| 自检清单 | 只读摘要 + 「去生成」 | 未满足必填项时禁用并提示缺哪项 |
| 等待 | 进度文案、warnings 折叠区 | 失败 → 回备素材；过期 → 回生成 |
| 选候选 | 缩略图、`confirm` | 返回修改 → 清空 job 状态回到步骤 2～3 |
| 落画布 | 「新增图层」必选；可选「替换选中层」 | P1 替换 |
| 保存 | 指向主编辑器保存按钮的短提示 | 无 |

---

### 4.2 备选方案：**三步轻量 Wizard（紧凑版）**

**一句话**：不拆任务卡，用顶部 **Step 1/2/3** 横条：① 类型 + 素材 ② 文案与参数 ③ 结果与落画布。

- **优点**：开发量比任务卡略小，仍比现状多两步结构化。  
- **缺点**：多参考「逐张角色」在一步里易拥挤；复杂用户仍不如任务卡清晰。  
- **适用**：工期紧时先做横条三步，后续演进为任务卡。

---

### 4.3 四种能力在模型中的覆盖

| 能力 | 在主方案中的位置 | 与画布关系 |
|------|------------------|------------|
| **文生图** | 任务卡「只用文字」 | 默认新层 |
| **单参考编辑** | 任务卡「一张图改款」+ 可选 strength（若前后端接通） | 新层 / P1 替换选中层 |
| **多参考融合** | 任务卡「多张图」+ **每张角色检查点** | 新层 |
| **迭代** | 任务卡「基于上一张」：引用上一 confirm 的 URL 或当前选中图片层为 `reference_asset_ids` 首图 | 默认新层（保留历史层便于对比） |

## 5. 与 API / 数据契约对齐

以下字段定义以 `docs/AIGC-API-MVP-SKELETON-v2.md` 与 `lib/aigc-provider-contract.ts` 为准。

### 5.1 `POST /api/aigc/generations` 字段映射

| 字段 | 交互方案中的来源 | MVP 可落地 | P1 / 备注 |
|------|------------------|------------|-----------|
| `product_id` | 当前商品 | ✅ 已有 | — |
| `mode` | **用户选任务卡后显式写入**（不再仅靠隐式推断） | ✅ 已有字段；需 UI 显式 | 减少误用 multi_ref |
| `prompt` | 文案框 | ✅ | txt2img 必填 |
| `negative_prompt` | 负面词框 | ✅ | |
| `candidate_count` | 滑块 1～4 | ✅ | |
| `reference_asset_ids` | 上传经 `reference-assets` 后；或「从画布导出再上传」的 asset 列表 | ✅ | 迭代：首 id 可为「上一输出」 |
| `references` | 多参考任务卡里**用户为每张图选 role** | ✅ 已有；需 UI 编辑数组 | 替代「轮换写死」 |
| `strength` | 单参考「改款强度」滑块 | ⚠️ 契约已有；**前端当前未传**，属小改动 | 建议 MVP+ 或标 P1 |
| `aspect_ratio` | 下拉（与 TTAPI 支持集合对齐） | ⚠️ 契约有；mock 忽略 | P1 体验增强 |
| `seed` | 高级折叠区 | 可选 | P1 |
| `composition_mode` | 多参考高级选项 `blend` / `collage_guided` / `controlnet_stack` | mock 仅记录；真实映射依赖 provider | **P1** 与模型能力表对齐后再暴露 |

### 5.2 其它端点（不变）

- `POST /api/aigc/reference-assets`：仍 multipart `files`。  
- `GET /api/aigc/generations/[jobId]`：轮询；展示 `warnings`。  
- `POST .../confirm`：仍 `candidate_index`；落画布逻辑保持「写入图层 + 提示保存定制」。

### 5.3 Provider 能力边界（一句话）

- **TTAPI（Flux）**：文生图与带参考的 edits 已接；多图时以厂商 **edits** 能力为准，复杂 composition 可能需 warnings 或降级为「近似融合」。  
- **DashScope / fal**：同样走统一 `AigcProviderGenerateInput`；具体是否支持多参考、是否忽略 `composition_mode` 以各 `lib/aigc-providers/*` 实现为准。  
- **mock**：用于 CI 与 demo，多参考语义简化，**产品演示勿把 mock 效果当真模型承诺**。

## 6. 验收标准（用户能自己完成）

以下为 **5～8 条**可测句子（黑盒验收口吻）；不要求用户懂 API 名。

1. 用户在不看文档的情况下，能说出「我这一次是在：只写字 / 用一张图改 / 用多张图一起」中的哪一种，且界面显示与其实际选择一致。  
2. 选择「多张图一起」时，用户能为**每一张**参考图指定角色（主体/风格/版式/其他）后再生成，且生成前能看到**只读摘要**并需确认才能点生成。  
3. 文生图路径下，未填主要文案时，界面阻止生成并提示缺什么（与后端 `txt2img` 必填一致）。  
4. 生成完成后，用户能在倒计时结束前完成「选一张 → 确认」，且确认后能在画布上看到新图；用户能区分「图已上画布」和「定制已保存到服务器」两个状态（文案或步骤上有明确区分）。  
5. 用户对候选不满意时，能回到「改文案或改参考」的步骤重新发起一次任务，而无需刷新整页。  
6. （P1）用户能把「当前画布选中的图片层」一键作为下一次图生图或多参考的输入，无需手工下载再上传文件。  
7. （P1）用户能选择「替换当前选中图层」而非总是新增一层，且该操作前有确认。  
8. 若服务端返回 `warnings`（如模型降级），用户能在界面上读到至少一句可读中文说明，而不仅仅是一张「莫名其妙的图」。
